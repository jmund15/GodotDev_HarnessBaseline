`read_files` / `read_web` reliability is governed by **OUTPUT size, not input file count**. Claude Code stops inlining a tool result and spills it to a `tool-results/*.txt` file once the result grows past **~30–40K characters** — the "result saved to a file, now unwrap the `{result: "..."}` JSON" pain. Measured 2026-05-31 by bisection: a ~28.5K-char result arrived inline; a ~43K-char result (worker output complete, verified by an end-of-output sentinel — so NOT model-truncated) spilled to disk; historical spill files cluster 58K–126K bytes.

This is a **Claude-Code-side limit** on the result it receives, independent of the worker: **raising `max_tokens_reader` does NOT prevent spilling** — a result completing under the worker cap still spills if it exceeds Claude Code's inline cap.

**The second wall — `max_tokens` TRUNCATION — is closer than it looks for structured data (corrected 2026-06-02).** The worker stops generating when it hits `max_tokens_reader`, returning `finish_reason=length` and a GENUINELY INCOMPLETE result — distinct from spilling (a complete result that's merely large). The char-distance to this wall depends on tokenization: prose is ~4 ch/tok, but **dense structured extraction (markdown tables, IDs, JSON) is ~1.2–1.5 ch/tok**, so the old 32K-token cap truncated at only **~40–50K chars — right on top of the spill zone**, not 5× past it. A `roadmap_next`-style raw extraction of 9 roadmaps hit truncation almost every call. Fixes (2026-06-02): `max_tokens_reader` raised 32K→**131072** (DeepSeek ceiling 384K), and the spill manifest is now **truncation-aware** — a truncated large result returns an `INCOMPLETE — CUT OFF mid-output … reduce output volume` manifest, NOT the `SUCCESS / do NOT rerun` one (which previously buried the `[TRUNCATED]` prefix and told the agent to trust a partial file).

**Why:** Two recurring misdiagnoses both fail. "Raise the cap" can't help — spilling is Claude-Code-side and fires far below the worker cap. "Limit input to <5 docs" targets the wrong variable — a bounded digest over 9 docs stays inline (~1KB despite ~29KB input); a verbatim extraction of one ~43KB doc spills. Cheap reader models also inflate verbatim output unpredictably, so even input bytes only loosely predict output.

**How to apply:** Before any `read_files`/`read_web` call, ask: *is the output bounded (a focused answer/digest) or unbounded (raw/verbatim extraction that grows with input)?* Bounded → many files fine, no batching. Unbounded → **batch so each call's output stays under ~28K chars (≈1 medium source file)**, OR re-scope to a structured/bounded digest (compression is the tool's whole value). Large results (>~25K chars) now auto-return as a manifest pointing to a clean raw file — but **read the manifest's framing**: `SUCCESS` = complete, read the file (Read offset/limit or grep/jq), do NOT rerun; `INCOMPLETE` = worker truncated, the file is PARTIAL, **reduce output volume** (split into fewer files/call, narrow the question, or request a bounded digest) and verify completeness (e.g. expected entry count). **Switching models does NOT help** — `max_tokens` is global per read call (kimi gets the same cap, and its hidden reasoning leaves *less* room). When output ≈ input (verbatim reproduction), skip the worker and Read the files directly (same token cost, no round-trip).
