# Doc Command: Before Writing

<!-- Single source of truth for pre-writing setup steps shared by all doc commands. -->
<!-- Referenced by: /doc_usage, /doc_architecture, /doc_retrospective, /doc_npc, /create_obsidian_design_doc -->

## 1. Identify the System
Determine the system name from the user's request. This becomes `{SystemName}` for folder/file paths.

## 2. Locate the Doc Folder
Folder-naming rules: `agents/documentation_structure.md`. Vault tooling: `obsidian_conventions` skill.
- Normalize the system name (PascalCase, no spaces)
- Use `Glob` to check whether the doc folder exists

## 3. Check for Existing Doc
- `Read` the target document path if it exists (it's a normal vault filesystem path).
- **If it exists:** You are UPDATING, not rewriting. Read the full doc, identify what has changed based on the current codebase, update those sections, and append a changelog entry. Preserve unchanged content.
- **If it doesn't exist:** Create from scratch.

## Reason, Then Delegate — Writing via `write_doc`

Per the Documentation Delegation Rule (HARD), doc prose is generated by `mcp__ai-worker__write_doc`, **never typed directly by you**. After the deep-read, decide what the doc must SAY — structure, claims, facts — then build a spec and call the tool. The reasoning is the irreplaceable step; the worker has none of your codebase context.

**Build the spec** (one string passed as `spec`). The linter matches four LITERAL headers (case-insensitive substring) — include all four verbatim or the call is rejected:
- `Outline` — section headings in order, taken from this command's Document Structure block.
- `Key claims` — the 3–10 specific things the doc must communicate.
- `Voice/tone` — supplied by the calling command (e.g. architecture → terse-technical; usage → instructional second-person; retrospective → first-person candid; design → terse-technical).
- `Must-include facts` — exact class names, file paths (`Foo.cs:42`), export names, dates/commits. The worker invents nothing; anything absent from the spec will not appear.

Scrub conversational provenance from the spec body (`per user direction`, `you mentioned`, `user surfaced`, …) — the worker reproduces spec phrasing verbatim (`feedback_write_doc_spec_provenance_discipline.md`).

**Call:**
`mcp__ai-worker__write_doc(doc_type="<architecture|usage|retrospective|design>", doc_path="<absolute vault path>", spec="<spec>", reference_files=[<existing doc, when updating, for voice/structure parity>])`
- `doc_type` selects the worker system prompt. The `obsidian` modifier **auto-applies** because `doc_path` is under `ObsidianVault` — do NOT request it. Confirm the return payload shows `<type>+obsidian`.

**Claude owns every merge — the worker only ever generates NEW prose.** `write_doc` regenerates the whole file from the spec; it has no append mode. So never hand the worker a file whose content must be preserved:
- **New doc:** spec carries the full structure; let `write_doc` write the whole file.
- **Existing doc (update):** `Read` it, identify the changed sections, call `write_doc` with an `Outline` of *only* the changed headings to generate just those sections' prose, then `Edit` them into the existing file and append the changelog row yourself. Untouched sections never pass through the worker.
- **Append-only history (retrospective):** the worker generates *only* the new dated entry; you append it via `Edit`. Prior entries never enter a regen.

**Verify:** read the returned headings/structure block — NOT the prose. Confirm the section headers match your Outline.

## Shared Rules
All Obsidian formatting, wikilink, and callout rules are auto-injected worker-side by `modifier.obsidian.md` (path-triggered) and documented in the `obsidian_conventions` skill (auto-loads on vault work). The command's callout-type lists are guidance for the spec, not something you hand-format.
